# CodeNexus 设计哲学

## 为什么收缩

一开始把仓库做成“大而全代码货架”看起来很热闹，但很容易长成两种东西：

1. 主题老化很快的旧题材样板
2. 自我感动的分类学

这两种东西都会让真正有价值的研究工作变钝。

所以现在 CodeNexus 的核心不是“存多少代码”，而是“如何更清醒地研究代码”。

## 核心信条

### 1. 证据 > 包装

先收集事实，再给动作建议。  
不要用漂亮目录、伪精确分数、或者热词包装理解。

### 2. 删除 > 美化过时主题

当一批主题已经明显老化、失去牵引力时，优先删掉。  
不要通过改名、换分类、加新壳继续维持它们的存在感。

### 3. 协议 > 样板

真正能复用的往往不是整坨代码，而是：

- 怎么分析
- 怎么验证
- 怎么判断值不值得提炼
- 怎么阻止自己发水 PR

这些东西更适合做成 prompt、protocol、rubric 和脚本。

### 4. 运行真相 > 测试绿灯

没有 runtime truth，就不要高估自己的理解。  
测试可以帮助确认边界，但测试本身不是现实。

### 5. 当下问题 > 旧时代目录

优先关注这些方向：

- agent workflows
- retrieval / memory / context
- browser / tool use
- automation / eval / orchestration
- 真实协作和运行约束

而不是让仓库默认围着一堆过熟、过泛、过旧的题材打转。
