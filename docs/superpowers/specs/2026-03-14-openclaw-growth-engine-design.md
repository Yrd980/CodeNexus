# OpenClaw Growth Engine Design

**Date:** 2026-03-14  
**Status:** Draft for approval  
**Owner:** Codex

> This design is authored in the CodeNexus repository as the product-level source of truth, but it targets the future implementation work in the OpenClaw repository.

## 1. 背景

OpenClaw 现在已经有一条基本投喂链路：

1. 发现 GitHub 项目
2. 分析项目
3. 提取精华代码
4. 可选地向原仓库提 PR
5. 向 CodeNexus 投喂模块

这条链路已经足够启动，但还不够“现实导向”。如果继续只追求“找到仓库 -> 提取代码 -> 写测试 -> 合并”，OpenClaw 很容易退化成高噪音搬运工：

- 追逐热度，而不是追逐可迁移的工程认知
- 提交低价值 PR，只为了留下贡献记录
- 用容易通过的 happy path 测试，掩盖对问题理解不足
- 把 Docker、边界测试数量、语言偏好误当成质量本身
- 每次都像第一次分析项目，无法随着反馈变得更聪明

OpenClaw 下一阶段要进化成一个长期运行的“代码研究员”：

- 从现实世界持续发现值得学习的项目
- 从代码中提取能服务 Startup 的通用思想
- 只在真正有价值时向原项目提 PR
- 把认知沉淀为 CodeNexus 模块、知识卡片和审核标准
- 根据实际结果更新自己的判断模型

## 2. 目标

### 核心目标

1. **高信号发现**  
   以 GitHub Trending 为起点，持续发现对 Startup 工程实践有迁移价值的项目。

2. **高价值理解**  
   输出的不只是“这段代码能跑”，而是“它解决了什么问题、为什么这样设计、边界在哪、何时不该抄”。

3. **高价值贡献**  
   只在能帮助维护者、改善用户认知、修复真实设计问题时提 PR。

4. **高质量沉淀**  
   对语言保持开放，但对沉淀结果保持严格：进入 CodeNexus 的内容必须可运行、可解释、可测试。

5. **持续成长**  
   OpenClaw 必须根据 PR 接受率、提炼内容后续价值、误判案例持续修正自己的策略。

### 非目标

- 不追求每天固定 PR 数量
- 不追求覆盖所有编程语言的完整实现能力
- 不把 Docker、脚手架完整度、边界 case 数量当成核心质量指标
- 不把“测试全绿”当成“理解真实问题”的替代品

## 3. 设计原则

### 3.1 现实优先

GitHub Trending 是最简单、最低摩擦的发现入口，所以应该作为第一阶段默认候选池；但 Trending 只是入口，不是价值证明。OpenClaw 必须做第二层判断：这个仓库到底有没有值得迁移到 Startup 场景的工程思想。

### 3.2 价值优先

OpenClaw 提 PR 的目标不是“留下爪印”，而是“留下维护者会感谢的改动”。  
优先级从高到低：

1. 修正错误设计或错误认知
2. 修复真实可复现的缺陷
3. 改善关键文档中误导性的做法
4. 提升可配置性、可维护性、可迁移性
5. 最后才是边界细节和局部打磨

### 3.3 第一性原理验证

测试不是表演。OpenClaw 的验证必须围绕“这个实现是否真的成立”展开，而不是围绕“这个测试是否容易通过”展开。

每次提炼或 PR 都至少回答三个问题：

1. 它在最小真实运行路径上能不能成立？
2. 它依赖的关键假设被打破时，会怎样失败？
3. 它被迁移到陌生上下文后，接口和边界还清晰吗？

### 3.4 语言开放，沉淀统一

OpenClaw 不限制源仓库语言。Go、Rust、TypeScript、Python、Ruby、Elixir 都可以学习。  
但沉淀出来的内容必须统一成两种产物：

- **认知产物**：设计洞察、反模式、知识卡片、伪代码
- **工程产物**：可运行的 CodeNexus 模块、测试、README、`.meta.yml`

### 3.5 成长优先于吞吐

OpenClaw 应该越来越会判断：

- 什么仓库值得研究
- 什么代码值得提炼
- 什么 PR 值得发
- 什么测试是“理解”，什么测试只是“取巧”

系统必须显式记录这些判断的后果，然后把经验回灌到下一轮。

## 4. 系统闭环

OpenClaw 的推荐闭环如下：

```text
发现候选仓库
  -> 仓库评分
  -> clone / update
  -> 仓库理解
  -> 提取高价值模式
  -> PR 机会评分
  -> 选择动作
     - 向原仓库提 PR
     - 向 CodeNexus 投喂
     - 只记录认知
     - 丢弃
  -> 运行结果回写
  -> 更新策略与认知
```

这里最重要的不是“自动化更多动作”，而是“在每个动作前做更好的判断”。

## 5. 核心子系统

### 5.1 Discovery Engine

#### 输入

- GitHub Trending 日榜/周榜
- 近期 release 活跃项目
- 历史高质量维护者的项目网络
- 已验证高质量仓库的相邻仓库

#### 第一阶段实现

先只做 Trending：

- 默认抓取 24 小时内最热仓库
- 维护一个回溯窗口，逐步向前扫更早的 trending 样本
- 为每个候选仓库记录首次发现时间、语言、stars 增速、主题标签

#### 关键要求

- 不因为“流行”就进入深度分析
- 要能去重，避免反复研究同一仓库
- 要能记录“为什么这个仓库被跳过”

### 5.2 Repository Intelligence Engine

这个子系统负责判断“仓库值不值得认真研究”。

它还负责识别仓库的**语言画像**和**运行画像**：

- 主要语言与混合语言比例
- 构建命令、测试命令、最小启动命令
- monorepo / polyrepo 结构
- 是否需要额外服务才能验证关键路径

#### 核心评分维度

1. **Startup Relevance**  
   是否解决 auth、payments、infra、realtime、workflow、AI integration、dev tooling 等真实问题。

2. **Transferability**  
   代码和设计是否能从原项目中剥离出来，迁移到别的业务中。

3. **Design Depth**  
   是否包含值得学习的边界设计、错误处理、配置抽象、并发模型或工程组织方式。

4. **Operational Reality**  
   是否真的在解决真实工程问题，而不是 demo、玩具或模板噪音。

5. **Maintenance Signal**  
   issue / PR 讨论质量、release 节奏、维护者响应、测试与文档是否体现认真维护。

#### 输出

- `research`
- `extract-only`
- `pr-opportunity`
- `skip`

### 5.3 Extraction Engine

OpenClaw 不应该直接“复制代码段”，而应该先做结构化理解：

- 它解决的核心问题是什么
- 它依赖哪些前提条件
- 它的抽象边界在哪里
- 哪些部分是项目私有逻辑
- 哪些部分可以一般化
- 哪些部分看起来聪明，但其实不值得沿用

#### 提取产物

1. 结构化分析 JSON
2. 候选 CodeNexus 模块草稿
3. 候选知识卡片
4. 反模式记录

### 5.4 Contribution Engine

这是 OpenClaw 最容易走偏的地方，所以要单独强调。

#### 高价值 PR 判定规则

只有满足以下至少一项时，OpenClaw 才进入 PR 设计阶段：

- 能修复真实 bug，并且有清晰复现路径
- 能消除会误导使用者的错误文档或错误示例
- 能改进明显不合理的抽象边界、配置设计或错误处理
- 能补上缺失的关键可维护性能力，而且不会引入额外复杂度

#### 明确不推荐的 PR

- 只为了刷存在感的微小命名调整
- 缺乏实际影响的边角 case
- 维护者未表现出需求的过度抽象
- 脱离项目风格的“理论上更优”重写

#### PR 评分

PR 机会应包含以下评分：

- `user_impact_score`
- `maintainer_acceptance_score`
- `proof_strength_score`
- `change_surface_score`
- `brand_risk_score`

只有在高影响、高证据、低品牌风险时才执行。

如果变更面较大、证据不足、或维护者语境不清晰，则不直接提交，而是进入 `draft-for-human-review` 队列。

### 5.5 Distillation Engine

进入 CodeNexus 的内容要同时满足：

- 保留源项目的有效设计思想
- 去掉源项目特有业务逻辑
- 让 Startup 团队可以复制、运行、理解

#### 推荐输出类型

1. **Runnable Module**  
   适用于 auth、queue、cache、notifications、search 等。

2. **Starter Pattern**  
   适用于 API starter、CLI starter、browser extension starter 等。

3. **Knowledge Card**  
   当设计洞察很有价值，但不适合沉淀成完整模块时使用。

4. **Pseudo-code Pattern**  
   当语言差异太大，但思想极强时，先保留伪代码和结构说明。

### 5.6 Verification Engine

这是 OpenClaw 需要明显升级的地方。

#### 现有问题

- 容易围绕 happy path 写测试
- 容易验证“形式正确”，不验证“现实成立”
- 容易把单元测试通过误当成设计理解完成

#### 新验证标准

每次提炼和 PR 都尽量覆盖以下三层：

1. **Executable Truth**  
   按 README 或最小启动方式真实跑一次，确认不是纸上模块。

2. **Assumption Break Test**  
   故意打破一条关键假设，观察错误是否暴露在正确边界。

3. **Portability Test**  
   把模块放到更小、更陌生的上下文里验证接口是否仍然自洽。

#### 测试哲学

边界测试本身不是价值。  
只有当边界反映了真实风险、真实误用方式、真实失败模式时，它才值得存在。

### 5.7 Learning Engine

OpenClaw 必须具备“复盘记忆”。

#### 要记录的反馈

- 哪些仓库分析后发现其实没价值
- 哪类 PR 经常被接受 / 拒绝
- 哪些提炼模块后来被证明不够通用
- 哪些测试通过了，但实际理解仍然出错
- 哪些 prompt / rubric 导致误判

#### 学习回路

每个周期结束时，生成一份认知更新日志：

- 本轮发现了什么新的判断标准
- 本轮哪些旧判断被推翻
- 需要更新哪些 prompt、评分规则和审核清单

### 5.8 Runtime and Operations

OpenClaw 的运行方式应该默认面向长时间无监督执行。

#### 运行模型

- 24h 调度循环
- 有明确批次边界，避免无限漂移
- 每一批次结束后做 checkpoint
- 在空闲窗口做自更新和回收

#### 自更新要求

OpenClaw 需要自动 `git pull` 更新：

- 自身代码
- prompt / rubric / protocol
- 规则配置

但更新必须安全：

1. 等当前批次结束
2. 写入状态 checkpoint
3. `git fetch` + fast-forward pull
4. 跑最小自检
5. 成功则继续，失败则回滚到上一已知健康版本

#### GitHub 平台礼仪与预算

24h 运行不等于无节制抓取。OpenClaw 必须显式管理：

- GitHub API / 页面抓取频率
- clone / fetch 并发预算
- PR 草稿与正式提交预算
- 对同一维护者或同一仓库的打扰频率

长期运行必须是“持续有礼貌”，而不是“持续打扰”。

#### 仓库缓存管理

对 clone 下来的第三方仓库：

- 支持 `clone if missing`
- 支持周期性 `git fetch` / `git pull`
- 记录最后更新时间
- 支持 TTL、磁盘预算和优先级回收

## 6. 关键决策

### 6.1 为什么从 Trending 开始

因为它是最低摩擦的发现入口，可以最快验证 OpenClaw 的分析和筛选能力。  
先把最简单入口跑通，再扩展信号源，符合现实主义。

### 6.2 为什么不限制语言

真正有价值的是设计思想，不是语言本身。  
如果 OpenClaw 只研究 TypeScript，会错过大量值得学习的工程实践。

### 6.3 为什么 PR 数量不是目标

低价值 PR 会快速损害 OpenClaw 的品牌和判断力。  
OpenClaw 应该用“被感谢的概率”而不是“提交成功的次数”来衡量贡献价值。

### 6.4 为什么不能迷信测试绿灯

如果测试只是围着最容易通过的路径设计，它只能证明“我们会写让自己通过的题”，不能证明“我们真的理解了系统”。

## 7. 成功指标

### 一阶指标

- 被深度研究的仓库中，高价值仓库占比
- 提炼结果中，真正进入 CodeNexus 的比例
- 原仓库 PR 的接受率与正向反馈率
- 被标记为“低价值/误判”的案例数量变化

### 二阶指标

- CodeNexus 新模块后续复用率
- 知识卡片被引用或复用的频率
- 同类仓库再次分析时，判断速度和准确率是否提升

### 失败信号

- PR 数量很高但接受率很低
- 新模块越来越多，但复用价值下降
- 测试一直全绿，但后续 reviewer 仍频繁指出理解偏差
- 越跑越忙，但认知标准没有升级

## 8. 风险与缓解

### 风险 1：Trending 噪音太高

缓解：

- 用第二层评分过滤
- 为“跳过原因”做结构化记录
- 每周复盘被跳过和被误判项目

### 风险 2：PR 价值判断失真

缓解：

- 引入 PR 机会评分
- 先从文档更正、真实 bug 修复开始
- 严格限制“纯优化型 PR”

### 风险 3：测试仍然形式主义

缓解：

- 强制要求最小真实运行验证
- 设计 assumption-break 场景
- 将“验证了哪些关键假设”写入产出物

### 风险 4：24h 运行导致策略漂移

缓解：

- 每批次设定固定预算
- 每批次只允许有限数量的深度分析
- 自更新前后做 checkpoint 和最小自检

### 风险 5：语言开放但实际只会分析少数语言

缓解：

- 先实现语言画像与运行画像，而不是一开始追求所有语言深度支持
- 把“能理解项目结构”与“能提炼完整 runnable module”拆成不同能力等级
- 对暂时无法完整实现的语言，优先输出知识卡片与伪代码模式

## 9. 分阶段落地建议

### Phase 1：把判断力立起来

- 保持 Trending 单一入口
- 引入 repo score、PR opportunity score、extraction score
- 升级 prompt 和 protocol，使其明确反对水 PR 与取巧测试

### Phase 2：把运行闭环跑起来

- 做仓库缓存、状态记录、周期性 `git pull`
- 建立 24h 批次循环
- 为每一轮生成认知更新日志

### Phase 3：把成长能力固化

- 分析 PR 接受/拒绝样本
- 根据反馈自动调整评分阈值和 prompt
- 形成“哪些信号长期有效”的经验库

## 10. 最终结论

OpenClaw 不应该被设计成“自动化搬运热门 GitHub 仓库”的工具。  
它应该被设计成一个长期运行、现实导向、价值导向、会复盘、会成长的代码研究员。

它的核心竞争力不是：

- 跑了多少仓库
- 发了多少 PR
- 写了多少测试

而是：

- 它是否越来越会判断什么值得学
- 它是否越来越会判断什么值得改
- 它是否越来越会把代码提炼成真正能复用的工程认知

如果做到这一点，OpenClaw 才会真正成为 CodeNexus 的增长引擎，而不是一个热度追逐器。
